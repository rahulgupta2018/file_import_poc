# Payment File Import Automation - Detailed Requirements

## 1. Document Control
- Version: `1.2`
- Date: `2026-02-26`
- Status: `Draft for implementation planning (updated with customer pain points & treatment strategy, CBO platform-specific field schema and constraints)`
- Source: `docs/requirement_doc.docx` (normalized and expanded)
- Golden Reference Files: `golden_reference_files/BACS_v4.xml`, `golden_reference_files/Example File BACS Apr 23 LLOYDS 1.xlsx`

## 2. Objective
Reduce customer effort and failure rates in payment file imports by automating mapping, validation, safe auto-fixes, and reusable map publication for CSV, fixed-length text, and BACS XML (`pain.001.001.03`) workflows.

## 3. Problem Statement
Current import mapping is highly manual and repetitive:
- Users upload files, manually map fields, configure parsing rules, and iterate through errors.
- Validation failures often lack actionable diagnostics.
- Repeated corrections are not remembered effectively.
- Support/Ops intervention is required for non-standard or repeatedly failing files.

## 4. Customer Pain Points & Treatment Strategy

The following pain points were identified from customer feedback, support ticket analysis, and CBO operations team interviews. Each is mapped to a concrete treatment strategy and the system capabilities that address it.

| # | Category | Customer Pain Points | Treatment Strategy | Addressed By |
|---|---|---|---|---|
| 1 | **Mapping & File Structure Issues** | Invalid formatting, wrong date formats, sort codes losing leading zeros, incorrect header sequence, unknown payment type, XML/CSV structure issues | Automation + schema inference + auto-fix engine + clear inline error messages | §8.1 (File Intake & Detection), §8.2 (Schema Inference & Auto-Mapping), §8.4 (Safe Auto-Fix Loop) |
| 2 | **Data Validation Failures** | Invalid beneficiary details, IBAN/SWIFT issues, mandatory field gaps, type mismatches | Validation engine + agentic guided fixes with next-best-action prompts | §8.3 (Validation Engine — 85 rules), §8.4 (Auto-Fix with before/after preview), §8.5 (Missing field prompt flow) |
| 3 | **No Clear Error Messages** | "System error occurred", "import failed" — customers cannot diagnose failures or understand what to fix | Human-readable error explanations + line-level diagnostics generated by the AI Agent + explainability panel ("why this failed", "why this fix is suggested") | §8.3 (Structured JSON errors with location + reason + suggestion), §13 (Explainability panel, inline fix suggestions) |
| 4 | **Duplicate / Reused Header Sequence Numbers** | Clients reuse header sequence numbers across files → repeated rejection by CBO platform | Auto-detect reused sequences within 24-hour window + auto-generate or auto-increment unique header values with user confirmation | §8.1 (Sequence number uniqueness detection), Appendix E rule VR-HT-001 |
| 5 | **Difficulty Understanding Import History** | Users cannot locate detailed error breakdown, confusion between screens, no single view of what went wrong | Unified diagnostic panel + AI Agent surfacing history and failure reasons + pipeline progress stepper + searchable import history with status filters | §13 (History view, pipeline stepper, inline errors), §9.5 (Observability metrics) |
| 6 | **Perceived Delays / Missing Imports** | Imports not appearing immediately after upload, fear of creating duplicates, no visibility into processing stage | Real-time progress indicators (upload → parse → map → validate → approve → publish) + status messaging at each stage (processing / queued / complete / failed) | §13 (Real-time import status), §9.1 (Performance SLAs: <10s for map proposal, <5s for validation) |
| 7 | **Self-Learning** | Repeated user corrections at the same locations are not remembered by the system — users keep fixing the same things | Mapping Memory that persists successful patterns and fix outcomes + promotes high-performing rules + recommends reusable maps based on similarity and history | §8.6 (Reuse and Self-Learning), §11.1 (rule_scores, mapping_history entities) |

> **Traceability**: Each pain point above is addressed by one or more functional requirements in Section 8 and reflected in the UI/UX requirements in Section 13. The clickable prototype (`prototype/index.html`) demonstrates the treatment for all 7 categories.

## 5. Target Outcomes
- Reduce time-to-successful-import for first-time files.
- Increase first-pass validation success.
- Improve reuse of previous successful maps.
- Improve transparency of failures with line-level explanations and suggested fixes.
- Preserve user control with explicit approval before map publication.

## 6. Scope
### In Scope
- File upload and staging.
- Auto-map proposal with per-field confidence.
- One-click validation and structured diagnostics.
- Safe deterministic auto-fix loop.
- Human-in-the-loop confirmation and publishing.
- Mapping lineage and versioning.
- Reuse recommendations based on file similarity and historical success.

### Out of Scope (for this phase)
- Full ML model training pipelines.
- Autonomous publish without user approval.
- Support for payment formats beyond defined inputs/outputs unless explicitly onboarded.
- CHAPS payments (via IBAN or Sort Code/Account Number).
- Faster Payments.
- International payments (via IBAN, BIC, or Clearing Code/Account Number).
- Urgent and non-urgent euro payments.
- Inter-account transfers.
- Multi-payment-type CSV files (combined payment types in a single file).
- MT101 format files.

## 7. Current vs Future Workflow
### As-Is
1. User selects import type and uploads file.
2. User manually configures mapping and parsing formats.
3. System validates and returns failures.
4. User repeatedly edits map/source and retries.
5. Valid map is saved and import proceeds.
6. Exceptions escalate to Ops/Support.

### To-Be
1. User uploads file.
2. System infers schema and proposes map with confidence badges and preview.
3. User runs one-click validation.
4. System returns line-level errors and deterministic fix suggestions.
5. System performs safe auto-fix + re-validation loop.
6. User confirms and publishes map.
7. System stores map, lineage, and outcomes; recommends reusable maps next time.

## 8. Functional Requirements
### 8.1 File Intake and Detection
- Support input types:
  - Fixed-length `.txt` — including BACS Standard 18 positional format (see Appendix D).
  - CSV (max `0.5 MB`) — two variants:
    - **CBO H/D/C/T format**: Record-type-driven layout per Appendix C.
    - **Non-standard / ERP-export CSV**: Header-row-based tabular files with arbitrary column naming (e.g., payment-run exports from SAP, Oracle, etc.). See Appendix D.2 for an example.
  - BACS XML (`ISO 20022 pain.001.001.03`, max `6 MB`).
    - **Note**: The XSD namespace URI is `urn:iso:std:iso:20022:tech:xsd:pain.001.001.03`. The CBO documentation refers to this as `PAIN 001.001.003` but the actual schema namespace uses `.03`. Validation must use the correct namespace.
  - BACS template variants with `H/D/C/T` records.
- Entry count limits per file (CBO platform limits):
  - Single-debit Bacs payments: `1,250` entries.
  - Inter-account transfers: `500` entries.
  - All other payment types: `2,500` entries.
- Detect:
  - Delimiter, encoding, header row, file structure.
  - Record sequencing validity (`H` -> `D` -> `C` -> `T` where applicable).
  - **Header row type**: Distinguish between CBO `H/D/C/T` record markers (column A) and standard columnar headers (named columns in row 1). Both patterns must be detected automatically.
  - Fixed-length record format identification (e.g., BACS Standard 18 by record length and positional patterns).
  - Sequence number presence and uniqueness (must be unique within a `24-hour` rolling window).
- XML-specific intake rules:
  - Encoding must be `UTF-8 without BOM`.
  - XML declaration must use `standalone="yes"` (not `standalone="true"`).
  - Validate against `pain.001.001.03` XSD schema variant (namespace: `urn:iso:std:iso:20022:tech:xsd:pain.001.001.03`).
- Generate file fingerprint for similarity/reuse.

### 8.2 Schema Inference and Auto-Mapping
- Infer candidate mappings from:
  - Regex/pattern dictionaries.
  - Known date and currency formats — must recognize at minimum:
    - `YYYYMMDD` (CBO standard CSV).
    - `YYYY-MM-DD` (XML / ISO).
    - `DD.MM.YY` (ERP exports, e.g., SAP).
    - `DD/MM/YYYY`, `MM/DD/YYYY` (common UK/US variants).
    - `YYYY-MM-DDT:HH:MM:SS.sssZ` (XML datetime).
  - Mandatory field rules.
  - Account/sort code patterns (6-digit sort code, 8-digit account, `SSSSSS-AAAAAAAA` composite).
  - XML tag semantics.
  - Fixed-length positional definitions (BACS Standard 18 — see Appendix D).
  - Column header semantic matching for non-standard CSVs (e.g., `"Payment Name"` → Beneficiary Name, `"Payee Bank Branch"` → Sort Code, `"Payment Document Amount (LC)"` → Amount).
- Handle non-standard CSV files:
  - Detect and skip non-payment columns (e.g., row numbers, supplier codes, foreign currency amounts).
  - Identify when mandatory BACS fields are **missing from the source file** (e.g., debit account) and flag for user prompt.
- Produce an initial proposed map with confidence score per target field.
- Show side-by-side source-to-target preview before validation.

### 8.3 Validation Engine

> **Complete validation rules register**: See **Appendix E** for the consolidated, numbered list of all 85 validation rules across file types — designed for business review.

- Validate:
  - Structural schema compliance.
  - Mandatory fields (see Appendix A for BACS XML mandatory/optional field matrix).
  - Data types and numeric constraints.
  - Account/beneficiary formats (including IBAN/SWIFT where relevant).
  - Date and currency normalization correctness.
  - `pain.001.001.03` schema compliance for XML.
  - Empty-element tag enforcement (e.g., `<InitgPty/>`, `<Dbtr/>`, `<FinInstnId/>` must be present but empty).
  - Fixed-value field enforcement (e.g., `MsgId` = `"Commercial Banking Online"`, `PmtMtd` = `"TRA"`, `Prtry` = `"UKBACS"`).
  - Character limit enforcement per field (see Appendix A).
  - Conditional tag removal: when optional fields are omitted, their entire parent tag subtree must be absent (see Appendix A notes).
  - Sequence number uniqueness within `24-hour` rolling window.
- BACS-specific business rules:
  - Payment amount must be `>= 0.01`.
  - Payment amount must include a decimal point and have fewer than `18` significant figures.
  - Currency must use 3-letter ISO code (e.g., `GBP`), no commas in amounts.
  - Beneficiary account number must be exactly `8` digits.
  - Beneficiary sort code must be exactly `6` digits, no dashes.
  - Beneficiary name max `18` characters.
  - Debit account format: `sortcode-accountnumber` (e.g., `123456-12345678`).
  - Debit account reference (if provided): min `6` alphanumeric characters, max `18` characters, with at least `2` unique characters.
  - Value date format: `YYYY-MM-DD` (XML) or `YYYYMMDD` (CSV).
  - Value date must be at least `2 business days` after the import date.
  - Created date-time format (XML only): `YYYY-MM-DDT:HH:MM:00.000Z`.
  - CSV current date (H-record): `YYYYMMDD` format.
- Codified CBO error categories for pre-catch diagnostics:
  - **Beneficiary errors**: account not 8 digits, invalid character set/field length for name/reference, failed bank info retrieval, invalid account, missing address book entry.
  - **Header/trailer errors**: file header missing, trailer not last record, sequence number not unique, no debits/headers/trailers.
  - **Other errors**: amount below minimum, debit account reference rule violations, entitlement errors.
- Return structured JSON errors with:
  - Location (row/column/tag/record index).
  - Failure reason ("why it failed").
  - CBO error category (beneficiary / header-trailer / other).
  - Suggested deterministic transforms.

### 8.4 Safe Auto-Fix Loop
- Apply only approved deterministic transformations, such as:
  - Trim/whitespace normalization.
  - **Date reformatting** — convert between known input formats:
    - `DD.MM.YY` → `YYYYMMDD` (CSV) or `YYYY-MM-DD` (XML). Apply 2-digit year windowing (e.g., `00–49` → `2000–2049`, `50–99` → `1950–1999`).
    - `DD/MM/YYYY` → target format.
    - `MM/DD/YYYY` → target format (with ambiguity detection when day <= 12).
  - Currency symbol normalization (e.g., strip `£` / `$` prefixes, remove commas from amounts).
  - **Amount unit conversion**: Detect amounts in pence/cents (integer values in fixed-length files) and convert to pounds/decimal (e.g., `100` → `1.00`). Flag for user confirmation when ambiguous.
  - **Decimal point insertion**: Add `.00` to whole-number amounts where required.
  - Split/merge predictable composite fields.
  - Leading-zero preservation for sort/account fields.
  - **Ignorable column stripping**: Remove non-payment columns (row indices, supplier codes, foreign currency amounts) from mapping output.
- Re-run validation after each fix batch.
- Stop criteria:
  - Validation success, or
  - No safe transforms remaining, or
  - Retry threshold reached.
- Provide before/after preview and explain changes.

### 8.5 Human-in-the-Loop and Publish
- User must explicitly approve final map before publication.
- **Missing field prompt flow**: When mandatory BACS target fields are not present in the source file (e.g., debit account sort code + account number missing from an ERP payment-run CSV), the system must:
  1. Surface a structured prompt listing the missing fields.
  2. Allow the user to supply the values manually (one-time or as defaults for this map).
  3. Store user-supplied values as part of the map definition for reuse.
  4. Re-validate after user input before proceeding to publish.
- Publish payload must include:
  - Friendly map name.
  - Map definition + version.
  - File fingerprint.
  - Applied rules/transforms.
  - User-supplied supplemental field values (if any).
  - Validation summary/results.
  - Timestamps and actor metadata.

### 8.6 Reuse and Self-Learning (No ML Training)
- Persist and score:
  - User corrections.
  - Validator-approved fix outcomes.
  - Rule success/failure counters.
  - Mapping success by fingerprint and context.
- Promote high-performing rules; demote error-prone rules.
- Recommend reusable maps based on similarity and historical success rates.

### 8.7 Output Requirements
- Produce:
  - Converted CSV in golden BACS format.
  - Validated XML (`pain.001.001.03`) when required.
  - Persisted map definitions in registry.

## 9. Non-Functional Requirements
### 9.1 Performance and Scalability
- Upload-to-initial-map proposal target: `<= 10s` for standard files.
- Standard file size baseline: CSV `<= 0.5 MB`, XML `<= 6 MB`.
- Standard entry count baseline: single-debit Bacs `<= 1,250` entries.
- Validation response target: `<= 5s` for standard payload.
- Support concurrent imports per tenant with quota/rate-limit controls.
- Reject files exceeding size or entry count limits at intake with clear error messaging.

### 9.2 Reliability and Availability
- Service availability target (initial): `99.9%` monthly for user-facing import APIs.
- Idempotent retry handling for Pub/Sub-driven stages.
- Dead-letter queues for failed async events.

### 9.3 Security
- Enforce authN/authZ via gateway and service IAM.
- Encrypt data in transit (TLS 1.2+) and at rest.
- Principle of least privilege for service accounts.
- Tenant isolation for mappings, file content, and diagnostics.
- PII-sensitive fields must be masked in logs and UI diagnostics.

### 9.4 Auditability and Explainability
- Every proposed fix and published map must be traceable.
- Persist lineage and decision rationale for diagnostics and compliance review.

### 9.5 Observability
- Centralized logging, tracing, and metrics per service.
- Core metrics:
  - Import success/failure rates.
  - Auto-fix effectiveness.
  - First-pass validation rate.
  - Map reuse rate.
  - Mean time to successful publish.

## Appendix A: BACS XML (`pain.001.001.03`) Field Specification

> Source: Lloyds Bank CBO — *How to import Bacs payments in XML*
> Schema: ISO 20022 `pain.001.001.03`
> XSD Namespace: `urn:iso:std:iso:20022:tech:xsd:pain.001.001.03`
> **Note**: CBO documentation references `PAIN 001.001.003` but the actual XSD namespace and golden reference file use `pain.001.001.03` (without leading zero). Always validate against the `.03` namespace.

### A.1 File Structure Overview

BACS XML files consist of three hierarchical sections:

| Section | XML Element | Description | Cardinality |
|---|---|---|---|
| **Group Header** | `<GrpHdr>` | Identifies the file. Exactly one per XML. All tags required. None of the information is used as part of the payment. | `1` |
| **Payment Information** | `<PmtInf>` | Contains debit account and payment-level details. Can be repeated for multiple payment groups. Some tags optional. | `1..*` |
| **Transaction Information** | `<CdtTrfTxInf>` | Contained within `<PmtInf>`. Represents each beneficiary payment. Can be repeated for Bacs Multiple payments. Some tags optional. | `1..*` per `<PmtInf>` |

### A.2 Complete Field Matrix

#### Group Header (`<GrpHdr>`)

| # | Field Name | M/O | ISO Tag Path | Char Limit | Format / Rules | Notes |
|---|---|---|---|---|---|---|
| 1 | Message Identification | **M** | `<CstmrCdtTrfInitn>` > `<GrpHdr>` > `<MsgId>` | 35 | **Fixed value**: `"COMMERCIAL BANKING ONLINE"` | The golden reference file (`BACS_v4.xml`) uses ALL CAPS. The CBO text guide shows title-case (`"Commercial Banking Online"`). **Use the golden file value (ALL CAPS) as the canonical reference** and validate case-insensitively as a fallback. |
| 2 | Created Date and Time | **M** | `<CstmrCdtTrfInitn>` > `<GrpHdr>` > `<CreDtTm>` | — | `YYYY-MM-DDT:HH:MM:00.000Z` (e.g., `2018-12-16T14:30:00.000Z`) | Year-Month-Day + Time in UTC. |
| 3 | Number of Transactions | **M** | `<CstmrCdtTrfInitn>` > `<GrpHdr>` > `<NbOfTxs>` | 15 | Any numeric value up to 15 characters | Structural only — not used for payment info. |
| 4 | Initiating Party | **M** | `<CstmrCdtTrfInitn>` > `<GrpHdr>` > `<InitgPty>` | — | **Must be an empty self-closing element**: `<InitgPty/>` | Do NOT populate with data. |

#### Payment Information (`<PmtInf>`)

| # | Field Name | M/O | ISO Tag Path | Char Limit | Format / Rules | Notes |
|---|---|---|---|---|---|---|
| 5 | Payment Information Identification | **M** | `<PmtInf>` > `<PmtInfId>` | 35 | Free text identifier | Identifies XML in CBO; not part of payment details. |
| 6 | Payment Method | **M** | `<PmtInf>` > `<PmtMtd>` | — | **Fixed value**: `"TRA"` | Must be exactly `TRA`. |
| 7 | Payment Type (Local Instrument) | **M** | `<PmtInf>` > `<PmtTpInf>` > `<LclInstrm>` > `<Prtry>` | — | **Fixed value**: `"UKBACS"` | Must be exactly `UKBACS`. |
| 8 | Debit Account Reference | **O** | `<PmtInf>` > `<PmtTpInf>` > `<CtgyPurp>` > `<Prtry>` | 16 | Min 6 alphanumeric chars, max 18 chars; at least 2 must be unique | **Conditional removal**: If omitted, remove entire `<CtgyPurp>` and `<Prtry>` subtree. Examples: `ab123` = invalid (5 chars), `abc123` = valid; `111111` = invalid (1 unique), `111112` = valid. Golden file value: `"Debt Acc Ref"`. |
| 9 | Value Date | **M** | `<PmtInf>` > `<ReqdExctnDt>` | — | `YYYY-MM-DD` | Payment execution date. |
| 10 | Debtor | **M** | `<PmtInf>` > `<Dbtr>` | — | **Must be an empty self-closing element**: `<Dbtr/>` | Do NOT populate with data. |
| 11 | Debit Account Number | **M** | `<PmtInf>` > `<DbtrAcct>` > `<Id>` > `<Othr>` > `<Id>` | — | `SSSSSS-AAAAAAAA` (sort code `-` account number, e.g., `123456-12345678`) | Sort code: 6 digits. Account: 8 digits. Separated by dash. |
| 12 | Financial Institution ID | **M** | `<PmtInf>` > `<DbtrAgt>` > `<FinInstnId>` | — | **Must be an empty self-closing element**: `<FinInstnId/>` | Do NOT populate with data. |

#### Transaction Information (`<CdtTrfTxInf>`)

| # | Field Name | M/O | ISO Tag Path | Char Limit | Format / Rules | Notes |
|---|---|---|---|---|---|---|
| 13 | End to End Reference | **M** | `<CdtTrfTxInf>` > `<PmtId>` > `<EndToEndId>` | 31 | Free text reference | Required but not displayed in payment info. |
| 14 | Credit Amount | **M** | `<CdtTrfTxInf>` > `<Amt>` > `<InstdAmt>` | — | Decimal >= `0.01`, **no commas**. Currency as XML attribute `Ccy` using 3-letter ISO code. | Example: `<InstdAmt Ccy="GBP">1500.00</InstdAmt>`. £1,500 must be `1500.00` not `1,500.00`. |
| 15 | Beneficiary Sort Code | **M** | `<CdtTrfTxInf>` > `<CdtrAgt>` > `<FinInstnId>` > `<ClrSysMmbId>` > `<MmbId>` | **6** | Exactly 6 digits, **no dashes** | e.g., `123456` |
| 16 | Beneficiary Name | **M** | `<CdtTrfTxInf>` > `<Cdtr>` > `<Nm>` | **18** | Alphanumeric + valid character set | Name of the beneficiary account. |
| 17 | Beneficiary Account Number | **M** | `<CdtTrfTxInf>` > `<CdtrAcct>` > `<Id>` > `<Othr>` > `<Id>` | **8** | Exactly 8 digits | e.g., `12345678` |
| 18 | Beneficiary Reference | **O** | `<CdtTrfTxInf>` > `<RmtInf>` > `<Strd>` > `<CdtrRefInf>` > `<Tp>` > `<CdOrPrtry>` > `<Prtry>` | **18** | If included, must contain data (non-empty) | **Conditional removal**: If omitted, remove entire subtree: `<RmtInf>`, `<Strd>`, `<CdtrRefInf>`, `<Tp>`, `<CdOrPrtry>`, `<Prtry>`. |

### A.3 Conditional Tag Removal Rules

When optional fields are not provided, **the entire parent tag subtree must be removed** from the XML. Leaving empty parent tags will cause validation failure.

| Optional Field | Tags to Remove When Omitted |
|---|---|
| Debit Account Reference (#8) | `<CtgyPurp>`, `<Prtry>` |
| Beneficiary Reference (#18) | `<RmtInf>`, `<Strd>`, `<CdtrRefInf>`, `<Tp>`, `<CdOrPrtry>`, `<Prtry>` |

### A.4 Empty Element Tags (Must Be Present But Empty)

The following tags must appear in the XML as **self-closing empty elements**. Populating them with data will cause CBO rejection.

| Tag | Parent Context |
|---|---|
| `<InitgPty/>` | `<GrpHdr>` |
| `<Dbtr/>` | `<PmtInf>` |
| `<FinInstnId/>` | `<DbtrAgt>` |

### A.5 Fixed/Constant Values

| Field | Required Value | Context |
|---|---|---|
| Message Identification (`<MsgId>`) | `"Commercial Banking Online"` | Group Header |
| Payment Method (`<PmtMtd>`) | `"TRA"` | Payment Information |
| Payment Type (`<Prtry>`) | `"UKBACS"` | Payment Type Information |

## Appendix B: CBO Platform Constraints and Known Error Categories

> Source: Lloyds Bank CBO — *Import errors explained*, *Template files for importing payments*

### B.1 File Size Limits

| File Type | Maximum Size |
|---|---|
| CSV | `0.5 MB` |
| XML | `6 MB` |

### B.2 Maximum Entry Counts

| Payment Type | Max Entries Per File |
|---|---|
| Single-debit Bacs | `1,250` |
| Inter-account transfers | `500` |
| All other payment types | `2,500` |

### B.3 Sequence Number Constraint

- Every imported file must contain a **unique sequence number** (cell `C1` in CSV/fixed-length files).
- A file will be **rejected** if another file with the same sequence number was imported within the **past 24 hours**.
- The system must track recent sequence numbers per tenant and pre-validate uniqueness before submission to CBO.

## Appendix C: BACS CSV Canonical Field Layout (`H/D/C/T` Record Structure)

> Source: `golden_reference_files/Example File BACS Apr 23 LLOYDS 1.xlsx` (Lloyds Bank CBO BACS CSV Template)

### C.1 Overview

The BACS CSV file uses a **record-type-driven layout** with four record types: Header (`H`), Debit (`D`), Credit (`C`), and Trailer (`T`). The file must be saved as **CSV (comma-delimited)** before import.

**Record sequencing**: `H` → `D` → `C` (repeatable) → `T`

- For **Single Bacs**: one `D` record followed by one `C` record.
- For **Multi Bacs**: one `D` record followed by multiple `C` records (one per beneficiary).
- For **additional Single Bacs payments**: repeat the `D` → `C` pair.
- The `T` record must always be the **last row** in the file.

### C.2 Record Definitions

#### Header Record (`H`)

| Column | Field Name | M/O | Format / Rules | Notes |
|---|---|---|---|---|
| A | Record Type | **M** | `H` (fixed) | Must be literal `H`. |
| B | Current Date | **M** | `YYYYMMDD` | Date the file is created. |
| C | Sequence Number | **M** | Numeric, unique per user per 24 hours | Uniqueness enforced within a 24-hour rolling window. |

#### Debit Record (`D`)

| Column | Field Name | M/O | Format / Rules | Notes |
|---|---|---|---|---|
| A | Record Type | **M** | `D` (fixed) | Must be literal `D`. |
| B | Value Date | **M** | `YYYYMMDD` | Must be at least **2 business days** after import date. |
| C | Debit Account Reference | **O** | Alphanumeric, min 6 chars, max 18 chars, at least 2 unique characters | Optional — identifies payment in statement. |
| D | Debit Account Number | **M** | `SSSSSS-AAAAAAAA` (e.g., `123456-12345678`) | Sort code (6 digits) + dash + account number (8 digits). |
| E | *(Reserved — leave blank)* | — | Empty | Must be blank. |

#### Credit Record (`C`)

| Column | Field Name | M/O | Format / Rules | Notes |
|---|---|---|---|---|
| A | Record Type | **M** | `C` (fixed) | Must be literal `C`. |
| B | Payment Amount | **M** | Decimal >= `0.01`, must include decimal point, < 18 significant figures | No commas. Example: `1500.00` not `1,500.00`. |
| C | Beneficiary Name | **M** | Max `18` characters | Valid character set required. |
| D | Beneficiary Account Number | **M** | Exactly `8` digits (e.g., `12345678`) | Numeric only. |
| E | Beneficiary Sort Code | **M** | Exactly `6` digits (e.g., `123456`) | Numeric only, no dashes. |
| F | Beneficiary Reference | **O** | Max `18` characters | Optional — if provided, must contain data (non-empty). |

#### Trailer Record (`T`)

| Column | Field Name | M/O | Format / Rules | Notes |
|---|---|---|---|---|
| A | Record Type | **M** | `T` (fixed) | Must be literal `T`. Must be the **last record** in the file. |

### C.3 Example CSV (After Instruction Row Removal)

```csv
H,20260226,001
D,20260228,,123456-12345678,
C,1500.00,Beneficiary Name,12345678,123456,Payment Ref
T
```

| Aspect | CSV | XML |
